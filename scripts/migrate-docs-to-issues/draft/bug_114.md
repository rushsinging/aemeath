# Bug #114：Stop hook blocked 缺少显式 chat loop 停止状态表达

| 字段 | 值 |

| 优先级 | 中 |

## 症状

Stop hook blocked 时，runtime 会追加 system reminder 并继续下一轮，但宏观状态依赖 `continue` / `break` 隐式表达。用户容易理解为“LLM 已经完成输出后，Stop hook 再阻止停止没有意义”。

## 修正语义

Stop hook blocked MUST 阻止 chat loop 真正停止。LLM 已经尝试结束不等于 runtime 已经 `Done`；blocked 时必须把 Stop hook 要求反馈给 LLM，回到 `Running`，直到后续 Stop hook success 才进入 `Done`。

## 根因

chat loop 结束路径缺少显式停止状态表达。Stop hook blocked 的语义分散在 loop control flow 中，依赖局部 `continue` / `break` 判断维持行为，缺少可检查的状态边界和非法状态转移防护。

## 修复

1. 引入轻量手写 `ChatLoopFsm`，显式表达 `Running -> Stopping -> StopHookBlocked -> Running -> Stopping -> Done`。
2. 在 Stop hook blocked 后恢复 `Running`，只有 Stop hook success 才进入 `Done`。
3. 强化 Stop hook blocked system reminder，明确“不能结束 / MUST 先满足 Stop hook 要求”。
4. 增加 FSM guard：非法转移 `warn + count`，关键退出/恢复路径使用 `assert_state()` 校验状态。
5. 保持现有 HookRunner/provider/TUI 事件架构，不引入外部 FSM 框架，不做 stream-time Stop hook。
