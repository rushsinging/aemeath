# Bug #21 对话中粘贴内容直接进 input queue（应进 input area）

**状态**：✅ 已修复，用户已确认
**发现日期**：2026-04
**确认日期**：2026-04-29
**优先级**：高
**修复 commit**：待提交

## 症状

对话进行中（processing 状态）在 input area 粘贴文本时，粘贴内容同时插入到 input area，并立即作为新一条消息进入 input queue。

用户预期是：粘贴只把内容放进 input area 继续编辑，按下 Enter 后才进队列。

## 实际效果

- 粘贴一段长文本 → input area 显示该文本，但同时队列已经多了一条等待发送。
- 用户还没编辑或确认，内容已经“半发出去”。
- 若粘贴的是多行长文，编辑/补充后再 Enter，可能再多一条同样内容的队列消息，造成重复。

## 根因

Bug #17 的修复实现里，processing 态的 Paste 分支同时做了两件事：

> 文本粘贴插入 input area + 入 queued_messages queue；空粘贴尝试剪贴板图片；图片路径粘贴加载图片

正确语义应只做“插入 input area”。入队判断应与普通 Enter 路径保持一致，由用户主动 Enter 触发。

## 修复

- `update.rs` Paste 分支：移除 processing 态下的 `queued_messages.push_back(...)` 调用，只保留 `input_area.insert_str(...)`。
- 入队语义统一：所有“提交到 queue”必须经过 Enter 路径。
- 与 Bug #19 保持一致：ask_user 态下 Paste 也只插入 input area，Enter 时走 reply_tx。

## 验证场景

- processing 中粘贴一段文本 → input area 出现文本 → 队列不增加。
- 编辑后按 Enter → 队列增加一条，包含编辑结果，且不重复。
- 连续粘贴两次 → input area 拼接两段，队列仍为 0 条。
- ask_user 等待时粘贴 → 同上，只进入 input area。
