问题区域 — apps/cli 内部的"隐藏领域逻辑"

⇤

TEA 状态机与事件系统

App / Msg / State / Cmd / Runtime 形成完整的 TEA 架构。30+ Msg 变体、多 Widget 协同、异步 Cmd 执行——这不是简单的"视图渲染"。

  • App::update() 核心分支
  • Msg 路由与分发逻辑
  • Cmd 队列与副作用管理
  • 多 Widget 间状态同步
⚛

渲染管线 (output_area)

output_area 不是简单 println。它包含语言检测、markdown 渲染、diff 高亮、滚动缓冲区、流式增量更新——有独立的渲染领域模型。

  • 语言检测与语法高亮
  • Markdown → ANSI 转换
  • Diff 块识别与着色
  • 滚动缓冲区 & 视口管理
⇄

输入处理管线 (input_area)

input_area 处理斜杠命令解析、自动补全、多行输入模式、粘贴检测——已形成独立的输入领域模型。

  • 斜杠命令解析与补全
  • 文件路径自动补全
  • 多行输入状态机
  • 粘贴检测与批量处理
↻

消息流管线

LLM 响应 stream → token 累积 → block 拆分 → 增量渲染。这构成了一条完整的实时数据处理管线，每个阶段都有自己的领域规则。

  • Stream → Block 转换器
  • 增量渲染调度
  • 工具调用结果注入
  • 用户确认中断流程
⌚

Session 生命周期

Session 的创建、恢复、暂停、消息历史管理、上下文压缩触发——这些生命周期管理逻辑在 cli 中有具体的实现，不是简单的 API 调用。

  • Session 创建/恢复/保存
  • 上下文窗口管理
  • 压缩触发策略
  • 历史消息分页
⚒

Widget 协同

多个 Widget（output_area、input_area、status_line、help_popup）需要在运行时共享状态、响应统一事件、协调焦点——这本身就是一个小型 UI 领域。

  • 焦点管理系统
  • Widget 间事件传播
  • 状态行实时更新
  • 弹窗/模态叠加

方案对比 — apps/cli 架构定位

方案 A

保持现状

apps/cli 继续作为 Runtime 的 thin adapter。内部结构不做 DDD 级拆分，维持 TEA 架构下的 monolith。

优势
改动最小，当前 TEA 架构已运行稳定
团队无需学习新的 cli 内部结构
问题
40+ 文件单 crate 膨胀，违反文件行数限制
"薄入口"定位与实际复杂度不匹配
新增 TUI Widget 缺乏清晰归属
方案 B

独立 Bounded Context

将 apps/cli 内部拆分为 3 个独立 Context：Display（渲染）、Interaction（输入交互）、Orchestration（编排），作为 UI 层的 DDD 设计。

优势
清晰领域边界：渲染/交互/编排解耦
利于 #36 Server 复用渲染管线
可独立测试每个 Context
问题
过度工程化风险——UI 层是否需要 DDD
Context 间通信增加复杂度
与 TEA 架构的 fit 需要评估
方案 C

TEA 分层治理

保持 TEA 架构，按 display / interaction / event 三层组织目录与模块边界。不引入 DDD 概念，但达成结构化治理。

优势
保持 TEA 架构一致性，学习成本低
与 #50 CLI 目录整理自然衔接
改动可控，渐进式治理
问题
缺乏统一的领域语言
Border 仍不清晰，依赖可能缠绕
未来 #36 Server 复用受限

推荐方案细节 — 方案 B: 独立 Bounded Context

UI Domain — Bounded Context 划分

  • Display (渲染) output_area + widgets + render
  • Interaction (输入交互) input_area + completion + command
  • Orchestration (编排) run_orchestration + session + msg_flow
  • TEA Kernel (共享) App / State / Msg / Cmd / Runtime

与 #50 的衔接

  • #50 Step 1 收拢碎片文件 → 对应 Context
  • #50 Step 2 命名规范化 → Ubiquitous Language
  • 本计划 Step 3 Context 间 API 定义
  • 本计划 Step 4 依赖守卫 + lint 规则
