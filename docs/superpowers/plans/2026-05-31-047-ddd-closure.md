**Goal:** 把 `2026-05-31-047-ddd-target-alignment` 完成后仍未满足 `docs/feature/specs/047-ddd-redesign.md` 的三块硬约束补齐：(A) 架构守卫真正覆盖 §6.4.8 维度并能抓到越界；(B) §6.4.5 rule6 — 把 `agent/shared/src/tool.rs` 的行为/并发/时间逐出 shared kernel；(C) §6.4.2/§6.4.6 — 把 gateway 从薄 re-export 升级为真 OHS trait + composition `wire_<feature>()` 装配。

## 0. 收口前基线（对照 spec 的未达标项）

对照 `docs/feature/specs/047-ddd-redesign.md`，target-alignment 后实测仍未满足：

| spec 条款 | 现状 | 收口策略 |
|---|---|---|
| §6.4.8 守卫完整 | 缺 `check-rust-file-lines.sh`；`check-share-minimal-kernel.sh` 不检测 `async_trait`/`Arc<Mutex>`/`tokio::sync`/`CancellationToken`/`SystemTime::now`/`Uuid::now_v7`，导致 tool.rs 越界**漏网却报绿** | Phase A：补 file-lines 守卫；给 kernel 守卫加源码模式，先用 per-file exemption 把 `tool.rs` 登记为已知债务（守卫诚实可见、聚合仍绿） |
| §6.4.5 rule6 + 决策#20 | `agent/shared/src/tool.rs` 仍含 `Tool`/`AgentRunner` trait、`ToolContext`(Arc<Mutex>/CancellationToken/Semaphore/mpsc)、`SessionReminders`(SystemTime::now/Uuid::now_v7) | Phase B：行为/并发类型迁回所属 feature（Tool/ToolContext/AgentRunner→`tools`），`SessionReminders` 保留为纯 DTO 但剥离时间/uuid；移除 exemption 后守卫真绿 |
| §6.4.2 rule2 / §6.4.6 | gateway 是薄 `pub use crate::core/business::*`，无 trait；composition 仅 `Arc::new(AgentClientImpl)`，无 `wire_<feature>()` | Phase C：为有跨 feature 服务的 feature 定义 OHS gateway trait + `wire_<feature>()`，composition 增加逐 feature 装配文件 |
| 验收门禁 clippy | 3 个 `dead_code` 警告（provider）→ `-D warnings` 红 | Phase D：清理 dead_code，删空残留目录，过完整门禁 |

**关键设计前提（Phase B 的相邻耦合）：**
- `agent/features/project/src/business/worktree.rs` 反向消费了 `ToolContext`（`derive_path_from_branch`/`create_worktree`/`exit_worktree`/`workspace_context_from_tool_context` 等取 `&ToolContext`）。若直接把 `ToolContext` 迁到 `tools`，会产生 `project → tools` 反向依赖（spec 只批准 `tools → project`）。**因此必须先解耦 project，再迁 ToolContext。**
- `agent/features/hook` 只用自有的 `HookData::Tool(ToolHookData)`，**不**消费 `share::tool::{Tool, ToolContext, AgentRunner}`，故 Phase B 不新增 `hook → tools` 依赖。
- `AgentRunner` 被 `tools`（agent_tool 派生子代理）与 `runtime` 同时消费。它是 tools 需要的**端口**、由 runtime 实现，故归 `tools`（方向 `runtime → tools` 实现 tools 的 trait，无环）。

## 1. shared::tool 当前公开项与去向

`agent/shared/src/tool.rs` 公开项分类与目标位置：

| 公开项 | 性质 | 目标位置 |
|---|---|---|
| `ToolResult`、`ImageData`、`WorkingContext` | 纯数据 DTO | **保留** `shared::tool`（数据契约合规） |
| `AgentProgressEvent`、`AgentProgressKind`、`AgentToolCallProgress` | 纯数据 DTO | **保留** `shared::tool` |

> 迁移采用"relocate-out-of-kernel"：这些类型一旦离开 shared kernel、进入 `tools` feature，其持有的并发/行为即合规（rule6 只约束 kernel，不约束 feature）。

- [ ] **Step 1: 写守卫脚本**

`.agents/hooks/check-rust-file-lines.sh`：

```bash
#!/usr/bin/env bash
# 单个 .rs 文件不超过 400 行（含测试），扫描 apps/ 与 agent/，排除 target/.git/.worktrees
set -euo pipefail
ROOT="$(cd "$(dirname "$0")/../.." && pwd)"
LIMIT=400
violations=0
while IFS= read -r f; do
  lines=$(wc -l < "$f")
  if [ "$lines" -gt "$LIMIT" ]; then
    echo "  $f: $lines 行 (> $LIMIT)"
    violations=$((violations+1))
  fi
done < <(find "$ROOT/apps" "$ROOT/agent" -type f -name '*.rs' \
          -not -path '*/target/*' -not -path '*/.git/*' -not -path '*/.worktrees/*')
if [ "$violations" -gt 0 ]; then
  echo "Rust file-lines guard FAILED: $violations 个文件超过 $LIMIT 行。"
  exit 1
fi
echo "Rust file-lines guard OK."
```
