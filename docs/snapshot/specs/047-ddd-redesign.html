Feature #47 · 架构基线 · 目标态

用 DDD 重新
设计 Aemeath 把散落在技术分层里的领域概念，重新归位成清晰的边界

这不是一次重构提交，而是后续重构、评审、功能设计都要对照的那张"图纸"。

核心域 · The Core Domain
Agent Runtime

从你输入的一句话，推进成完整的 Agent 协作：构造上下文 → 调模型 → 跑工具 → 调度子代理 → 处理交互 → 判断何时停 → 产出结果。其余能力都重要，但都是为它服务的"配角"。

01

先统一说话的词

很多 bug 和争论，根子是同一个词每个人理解不同。先把核心结构的"嵌套关系"钉死——谁包含谁，一目了然。

Session你和 Aemeath 的一整段协作容器
Chat你按一次回车触发的"一整轮"处理
Agent LoopingChat 内部那个不停转的循环
Main Turn / Child Turn*某个 Agent 为一个目标干的一段活
ModelInvocation*一次模型调用
ToolExecution*一次工具执行
Task updates待办清单推进
AskUser pause/resume由 AskUserQuestion 工具触发，Runtime 接管暂停/恢复
Stop Hook · Final response停止判断 / 产出
Agent 是一份"配置化的执行者"——定义用哪个模型、什么角色、什么权限，不是模型本身、也不是一次运行。
SubAgent 是父 Turn 派出去的"分身"，权限不能超过父。
Task 是运行时的待办/规划状态，不是独立模块；它的持久化投影进 Storage。
AskUser 由 AskUserQuestion 工具触发（协议上是 tool_use→tool_result），但暂停/恢复归 Runtime 接管，不走普通工具管线；而权限 Ask 是另一条非工具路径（§07）。
02

边界：谁负责什么

把系统切成一圈圈"有主的领域"。核心域在最中间编排一切；支撑域各管一摊事实与规则；防腐/基础设施负责和外部世界打交道。

核心域 · 唯一编排者

Agent Runtime runtime

维护 Agent/Session/Chat/Turn/Task，在 Chat 内跑 Agent Looping，调度 Provider·Tool·Memory·Prompt 与 SubAgent，判断完成/失败/取消/被阻止。所有其它域都由它编排。

支撑域 · 各管一摊

Config shared/config

把 CLI/环境/项目/全局/默认配置，合成一份不可变快照，给各域消费。

Tool tools

把模型想调的工具、技能、斜杠命令，安全地校验·授权·执行·归一化。

Project project

提供 cwd/路径/worktree/git 这些"事实"——只给事实，不做授权。

Policy policy

判断"这个操作准不准"：actor/action/resource → Allow/Ask/Deny。

Audit audit

记录"谁、在哪、做了啥、为啥"，用 correlation id 串起全链路。

Memory storage

跨会话的长期记忆：检索·沉淀·置顶·提醒·范围隔离。

Prompt prompt

加载并合并 AGENTS.md/CLAUDE.md/guidance/系统提示，裁剪成材料。

Storage storage

把会话/记忆/任务/历史/成本落盘，支持恢复与查询。

防腐 / 基础设施 · 对接外部

Provider provider

把各家模型协议翻译成统一的内部事件；具体 SDK 调用归 shared/adapter。

Hook hook

在生命周期点上跑你配的外部脚本，解析 Continue/Block 决定。

External Adapters shared/adapter

filesystem/git/shell/web/MCP/terminal 等所有外部系统适配的归属地。

03

依赖只能朝一个方向

入口很薄，只认 SDK 契约；Runtime 是唯一能调动支撑域的人；支撑域默认只依赖最底层的 shared。箭头永不回头——这就是防止双向依赖的关键。

CLI / TUI→ packages/sdk · AgentClient→ runtime · 唯一编排者→ tools · provider · prompt · project · policy · hook · audit · storage→ shared
已批准的少数"横向"例外：tools→project（worktree 上下文）、tools→storage（复用 memory 持久化）。横向依赖必须无环、走对方 api、并在 spec 里登记理由——其余一律禁止。

05

两个 feature 要协作，走哪种关系

DDD 给跨边界协作分了几种"正规姿势"。先想清楚是哪种，再决定代码放哪——而不是随手 import。

关系类型什么时候用落在哪
OHS / 发布语言默认：跨 feature 的同步查询 / 命令supplier 的 gateway + contract，经 api 调
防腐层 ACLconsumer 想用自己的词汇，不吃 supplier 的模型consumer 内部 core/ 的包装层
共享内核两边都认同的极小通用类型shared/ids · types · errors · config
领域事件状态变了要通知别人（异步）contract/events + runtime 编排分发
Conformist必须原样接受外部系统模型仅对外部系统 · shared/adapter
06

入口为什么"薄" — AgentClient 遥控器

CLI/TUI 只认一个"遥控器"接口 AgentClient，完全不碰 Runtime 内部。要看状态拿"快照"，要知道变化听"变更通道"。

入口
CLI / TUI只渲染、只转发
契约↔无内部细节
唯一通道
AgentClienttrait（packages/sdk）
薄代理↔
实现
Runtime拥有全部初始化与编排
为什么用"快照"不用引用？ 直接把 &Session 递给 TUI，会被读写锁卡住、阻塞渲染帧。所以改成：拍一张快照（无锁、CLI 不知道内存布局）+ 一条变更通道（只响个铃 ChangeSet，TUI 收到再去拉对应快照）。会话、成本、任务、项目各对应一个标记位，精确刷新。
07

权限、Hook、审计 — 三件事别混

最容易混的一处：权限判定（内部安全模型）和 Hook 判定（你配的外部脚本）是两回事。系统要能明确表达——"权限放行了，但 hook 把它拦下了。"

PermissionDecision

  • 来源：内部安全模型
  • 归属：Policy
  • 输出：Allow / Ask / Deny
  • 影响 capability：是
  • 用于权限继承：是

HookDecision

  • 来源：用户配置的外部脚本
  • 归属：Hook
  • 输出：Continue / Block
  • 影响 capability：否
  • 用于权限继承：否
ToolCall→ Permission→ PreToolUse Hook→ ToolExecution→ PostToolUse Hook→ Audit
Audit 把三笔分开记：权限判定、hook 判定、最终结果——只记录事实，不参与判断、不阻止执行。例如 Stop 阶段架构检查没过，hook 仍可拦下"完成"，审计照实记。
08

几条钉死的设计决策

完整 24 条见正文 §10，这里挑最影响"代码该放哪"的几条。

1Runtime 是核心域，是唯一编排者，可依赖所有支撑域。
2Agent 是配置化实体，不保存单次执行状态。
3入口必须薄，只经 AgentClient 契约接入，不直连支撑域。
4支撑域默认只依赖 shared；横向依赖须登记批准。
5shared 内核只放数据契约，不放行为 / IO / 并发 / 时间。
6PermissionDecision 与 HookDecision 必须分离。
7feature 边界由架构守卫强制，不靠人工约定。
8DDD 定边界，COLA 管内部分层，不互相替代。
