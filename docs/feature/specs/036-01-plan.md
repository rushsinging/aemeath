# #36 多 Agent 框架 — Plan（实施计划）

> **DDD 设计参考**：[Multi-Agent 框架 DDD 设计](../../superpowers/specs/2026-05-20-multi-agent-ddd-design.md) — Bounded Context 划分、领域事件、聚合根设计以该文档为准。

本计划按 Sprint 从最小可用闭环逐步扩展到多 Agent 编排与自我进化。总体节奏对应原 spec 的 MVP 分层：v0.1 单 Agent + 白板最小闭环 → v0.2 多 Agent 编排 → v0.3 Evolver + RAG。

## Sprint 划分

### Sprint 0 — 基础设施

**目标**：可编译、可运行的空架子  
**依赖**：无

**产出**：
1. `share/` 目录：REST DTO / OpenAPI schema 共享脚本、`share/proto/common.proto` 仅保留兼容期通用类型（Empty、AgentType、IdempotencyOptions；不再新增 WatchRequest）
2. `server/` 骨架：axum HTTP/WS 监听 `3000`，CORS 配置，OpenAPI 导出脚本（aide + schemars）
3. `infra/mongodb/`：collection 创建脚本（workspaces / … / work_items / agent_runs / outbox_events / controller_leases）、索引定义。MongoDB 是业务状态真相源；不依赖 Change Streams。
4. `infra/redis/`：Redis Streams 本地配置、stream 命名约定和 consumer group 初始化脚本。

8. `share/openapi/sdk/ts` SDK 目录（空壳 + package.json）
9. TOML 配置装配器（assembler.rs）：能从 5 个 Main Agent toml 文件的顶层扁平字段与 [permissions] 段解析 RoleConfig
10. Sub-Agent 角色 TOML 骨架（planner/coder/tester/reviewer/designer，共 5 个）
11. 日志基建：`aemeath.log` + `panic.log` + `agent.log`

**涉及模块**：share/openapi, server, infra/mongodb, infra/redis, agents runtime, config

**验证标准**：工作区能完成一次全量编译；docker-compose up 启动 Server + Agents + MongoDB + Redis；Mongo 初始化脚本可重复执行且不会破坏既有 collection/index；Redis stream / consumer group 初始化脚本可重复执行。

**目标**：用户能通过 API 创建 Workspace / Chat，发消息，收到 WS 推送  
**依赖**：S0.5

**产出**：

7. 幂等：AddMessage 按 idempotency_key 去重
8. `infra/gateway/nginx.conf`：反向代理（/api → Server REST，/ws → WebSocket）

**当前落地状态**：已完成内存版 Workspace/Chat/ChatMessage store、REST Workspace/Chat/Message、GET messages 分页、PATCH Chat、REST AnalyzeMessage、WS `type + payload` 正式 payload、gateway compose 配置、idempotency_key 去重与 server 内部 EventBus 骨架。后续需要将 EventBus/WS 推送迁移为 MongoDB Outbox + Redis BoardEvent，补齐 MongoDB 持久化、RBAC 中间件和 Redis cursor 断线恢复语义。

**涉及模块**：`apps/server`（REST + WS handler + Application Service）、`packages/core`（domain event / DTO）、`packages/sdk`（REST/WS SDK）、`apps/agents`（Redis consumer 后续接入）、`infra/gateway`。

**验证标准**：通过 REST 创建 Workspace/Chat 并发送消息后，WebSocket 客户端能收到全量快照与新增消息更新；重复提交相同 idempotency_key 只产生一条 ChatMessage。

### Sprint 4 — 任务执行

**目标**：确认需求后，Executor 编排 Sub-Agent 执行任务，产出结果  
**依赖**：S3

**产出（后端）**：
1. Project/Task REST/Application Service（Project CRUD: Create/Update/Get/List/Assign/Accept/Resume/Retry/Complete/Block/Fail/Cancel/ConfirmCancel/ForceCancel；Task 子实体操作: CreateTask/GetTask/UpdateTask/CompleteTask/ListTasks/CancelTask/ConfirmCancelTask/ForceCancelTask/FailTask/StartRetry；写 MongoDB + Outbox）
2. REST：Project 列表/详情/Resume/Cancel、Task 列表/详情/更新/Cancel/Retry、Agent 列表；Project/Task 创建由 Confirm / Scheduler / Agent 应用服务触发，不暴露通用 REST 创建入口
3. Project/ProjectTask 状态机实现（含取消 → 036-04-spec-state.md Cooperative Cancel 协议、重试、崩溃恢复；⚠️ ProjectTask 为 Project 聚合子实体，级联操作在 Project Application Service 内以聚合内事务执行）
5. Scheduler 实现：reconcile Project 状态并创建 Executor WorkItem；Executor 通过 Redis WorkQueue 消费并 MongoDB lease claim，不依赖 Watch
6. Executor 实现：接收 Project → 按 Task 顺序/并行编排 → 唤起 Sub-Agent（Planner/Coder/Reviewer/Tester/Designer）→ 收集产出 → 产出 ProjectResult.summary
7. Sub-Agent 编排：每个 Task 对应一个 Sub-Agent 角色，Executor 按 allowed_tools + context_window 管理上下文
8. Agent 间通信：Executor → Sub-Agent 进程内异步调用（Sub-Agent 与 Executor 同进程，无独立部署）
9. Context 上下文管理：每个 Sub-Agent 独立 context_window，超限时触发压缩
10. BoardEvent 增量推送：Executor 每完成一个 Task 通过 Application Service 写回 result 和 OutboxEvent，OutboxPublisher 发布 Redis BoardEvent，Server WS gateway 推送增量
11. Worktree 隔离：Executor 为每个 Task 创建独立 git worktree；Task 达终态（completed/failed/cancelled）时由 Executor 清理 worktree；Executor 崩溃时由 Scheduler reconcile / WorkItem lease 过期恢复流程清理孤儿 worktree
12. Merge Lock：Project 级别乐观锁（MongoDB），并发串行化，崩溃恢复级联释放
13. Sub-Agent 角色 TOML 完整配置（planner/coder/tester/reviewer/designer，共 5 个）

**产出（前端）**：
F1. Project/Task 进度面板：树形结构展示 Project → Tasks → Sub-Agent 状态
F2. Task 详情：Sub-Agent 日志片段、产出摘要
F3. 取消/重试操作

**涉及模块**：executor/、scheduler/（WorkItem 创建与 lease 恢复）、planner/coder/reviewer/tester/designer（Sub-Agent）、server（REST/Application Service + RBAC 中间件）、ui/

**验证标准**：确认草案后能生成 Project/Task 并创建 Executor WorkItem，Executor 从 Redis WorkQueue 收到通知后通过 MongoDB lease 领取执行，Task 完成时 UI 实时更新结果；模拟 Executor 崩溃后，Scheduler 能通过 WorkItem lease 过期和 Agent presence 丢失释放 Project 并将非终态 Task 回退到 pending 重新分配；Executor/Assistant 使用 board_write + work_item_* scope 写入成功，无相应 scope 时被拒绝。

## P0 设计约束

### 故障恢复

#### Executor 崩溃恢复

```text
1. Scheduler 扫描 Redis presence TTL 过期或 WorkItem lease 过期（heartbeat_timeout_sec, 默认 30s）
2. 根据 lease_owner / assigned_executor_id 查 Project status: assigned / in_progress → 崩溃恢复路径
   - assigned    → 清空 assigned_executor_id，Project 回退 pending（此时尚无 Task 被分配，无需级联回退）
   - in_progress → 仅在崩溃恢复路径清空 assigned_executor_id，并将 Project 回退 pending 以便重新分配
   - blocked     → Scheduler 通知 Chat，等待用户决策
   - completed/failed/cancelled → 终态不回退
3. Project 崩溃恢复条件更新（防并发）:
   db.projects.updateOne(
     { _id: project_id, status: { $in: ["assigned", "in_progress"] }, assigned_executor_id: old_id },
     { $unset: { assigned_executor_id: "" }, $set: { status: "pending" } }
   )
4. 释放该 Executor 持有的 merge_lock（Executor 同时最多一个 Project，理论上仅一个 merge_lock；此处做防御性全量释放）
5. 通过 Project Application Service 级联回退该 Project 下由该 Executor 持有的非终态 Task：
   in_progress / in_review / retrying → pending（清空 assigned_executor_id）
   仅在 retrying → pending 时保留 retry_count + last_error（in_progress / in_review 首次尝试无 retry_count 可保留）
6. 重新分配给新 Executor 时写入原因字段（如 "reassigned_after_crash"）

约束：普通执行失败不会直接 Failed → Pending；Task 级自动重试走 retrying → pending 路径（retry_count 递增，不超过 max_task_retries）；Executor 崩溃恢复直接回退非终态 Task 到 Pending；显式人工重开才会从终态 Failed → Pending。
```

#### Task 重试

```text
ProjectTask 状态:
  pending → in_progress → in_review → (completed | failed)
  pending / in_progress / in_review / retrying → cancelled（用户取消 / Project 级联取消）
  in_progress → retrying → pending（Executor 正常自动重试：下一次 attempt 重新进入分配流程，形成明确重试循环）
  in_review → retrying → pending（review 阶段判需重试：如 Reviewer 发现需重大返工，跳过 in_progress 直接 retrying；retrying 计数与 in_progress→retrying 共享同一 retry_count 上限）
  in_progress / in_review / retrying → pending（仅 Executor 崩溃恢复回退；普通失败不走此路径；Retrying 崩溃保留 retry_count + last_error）

Executor 重启后:
  1. 查询自身 assigned_task（status=in_progress 或 retrying）
  2. 从 Mongo 加载 Task 上下文（description + related_message_ids）
  3. 重新执行
  4. Sub-Agent 返回 retry_needed → 不写 final result；写一条 ProjectTaskResult(status=retry_needed) intermediate attempt result，ProjectTask.status=retrying，然后重新执行
  5. 连续失败或 retry_needed 超过 max_task_retries（默认 3）→ 写 ProjectTaskResult(status=failed) final result → ProjectTask.status=failed → 通知 Chat

重试携带 retry_count + last_error，Sub-Agent 可根据失败历史调整策略。
```

#### Redis / WorkItem 断线恢复

```text
v0.1 Redis stream 定位：可靠通知层，不是领域真相源。MongoDB WorkItem / OutboxEvent / 聚合状态是最终恢复依据。

关键约束:
  - API Server 不为 WebSocket subscriber 使用进程内缓冲；断线期间通过 Redis BoardEvent stream 尝试补发。
  - Redis stream 允许裁剪；cursor 过旧、gap 或无法补发时，Server 发送 snapshot_required，UI 通过 REST GET /api/workspaces/:ws_id/board 覆盖本地 state。
  - Scheduler 重启后通过 ControllerLease 获取 workspace 控制权，然后全量扫描 pending / stale leased / running 超时 WorkItem、assigned/in_progress Project，以及 Redis presence 丢失的 Executor。
  - Agent 重启后通过 WorkItem lease 和 AgentRun attempt 恢复；终态 WorkItem 的重复 Redis message 直接 XACK。

重连策略:
  - WS 指数退避重连：1s → 2s → 4s → ... → max 60s
  - WS 重连携带 last_stream_id；补发失败则 snapshot_required
  - Scheduler reconcile 频率上限：`full_scan_rate_limit_sec`（默认 60s，配置见 036-02 scheduler.*）

白板 WebSocket 断线:
  - UI 显示连接状态（disconnected / reconnecting / connected）
  - 收到 snapshot_required 或补发失败后：REST GET /api/workspaces/:ws_id/board 全量拉取最新快照，覆盖当前 state
```

### 幂等策略

- 所有会产生写入副作用的 REST / Application Service 请求必须携带 `idempotency_key`，由 Server 在目标资源范围内去重。
- `ConversationService.AddMessage`（Application Service）：按 `workspace_id + chat_id + idempotency_key` 去重，重复请求返回第一次创建的 ChatMessage。
- `RequirementApplicationService.Analyze`：使用条件更新完成原子抢占，仅允许 `pending` Requirement 进入 `analyzing`，避免多个 Assistant 同时分析同一需求。
- `RequirementApplicationService.Confirm`：按 `requirement_id + idempotency_key` 去重，避免重复创建 Project/Task。
- Project/Task 状态更新使用条件更新（例如校验 `assigned_executor_id`、当前 status、attempt），避免崩溃恢复、重试、并发 Executor 写回互相覆盖。
- Redis `last_stream_id` 只用于 WS 断线补发，不承诺永久保留；无法补发时通过 `snapshot_required` + REST snapshot 恢复最终一致性。
