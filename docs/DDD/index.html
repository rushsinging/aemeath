架构的 三重奏

DDD × Hexagonal × Clean Architecture

领域驱动设计、六边形架构与整洁架构，常被并列谈论并非偶然—— 它们指向同一条信念：把业务规则置于中心，让所有依赖朝内收敛， 把数据库、框架与界面统统推向可替换的外围细节。

01 The Common Creed · 共同信条

一条法则，贯穿三种架构THE DEPENDENCY RULE

源代码的依赖只能由外向内。内层的领域，永远不该知道外层的存在—— 它不认识数据库，不认识 HTTP，也不认识任何框架。

三种架构用不同的词汇讲述同一个故事：整洁架构称之为「依赖规则 The Dependency Rule」，六边形架构靠「端口 Ports」把方向反转，DDD 则通过「Repository 抽象」让领域对基础设施一无所知。

它们共同的技术支点，是依赖倒置原则（DIP）：当高层策略需要低层细节时，不直接依赖它，而是依赖一个由高层定义、由低层实现的接口。方向，由此被扭转。

02 Domain-Driven Design · 领域驱动设计

用统一语言，为复杂领域建模提出者 · Eric Evans · 2003

DDD 关心的不是分层，而是如何思考业务。它主张让开发者与领域专家共用一套统一语言（Ubiquitous Language）， 并把这套语言直接编码进模型。它分为两条战线：管理宏观边界的战略设计，与打磨领域模型的战术设计。

Tactical Patterns · 战术建模构件

实体

拥有唯一标识、贯穿生命周期保持连续性的对象。相等由 ID 决定，而非属性。

值对象

无标识、不可变，由属性整体定义相等性。如金额、坐标、日期区间。

聚合

一组绑定的对象，作为数据修改的一致性单元。外部只能引用聚合根。

聚合根

聚合唯一的入口与守门人，负责维护整个聚合的业务不变量。

仓储

以集合的错觉封装持久化。领域只见接口，不见 SQL——依赖倒置的落点。

领域服务

承载不属于任何单一实体的领域逻辑，用动词命名，保持无状态。

领域事件

记录「已经发生」的业务事实，用过去时命名，驱动跨聚合与跨上下文协作。

工厂

封装复杂聚合的创建过程，确保对象一诞生便处于合法状态。

Strategic Design · 战略设计
# 限界上下文 Bounded Context

模型的意义边界。同一个「商品」在下单与仓储中含义不同——各自为政，互不污染。

# 统一语言 Ubiquitous Language

团队与代码共享的精确措辞，消除翻译损耗。语言变，模型即变。

# 上下文映射 Context Map

描绘上下文之间的关系与集成方式：防腐层、共享内核、客户-供应商。

# 子域 Subdomain

区分核心域、支撑域与通用域，把最强的建模火力投向真正创造价值之处。

03 Hexagonal · Ports & Adapters · 六边形架构

让应用与外部世界，通过端口对话提出者 · Alistair Cockburn · 2005

六边形架构把应用想象成一个被端口（Port）包裹的核心：端口是核心定义的接口， 适配器（Adapter）则在外侧把具体技术翻译成端口能听懂的语言。左侧驱动应用，右侧被应用驱动—— 把测试、UI、数据库放在完全对称的位置上，任意一个都可插拔替换。

◤ 驱动端口 · Driving Ports

被外界调用、驱动应用行动的入口。用例接口、Application Service 便在此暴露。适配器：Controller、CLI、定时任务、测试。

◢ 被驱动端口 · Driven Ports

应用调用外界的出口，由核心定义接口、外层提供实现。Repository、消息发布、通知网关都在此接入——依赖被倒置。

04 Clean Architecture · 整洁架构

四层同心圆，一个方向的依赖提出者 · Robert C. Martin · 2012

整洁架构把系统画成四层同心圆，用一条铁律统摄：依赖只能指向圆心。 越靠内越稳定、越抽象、越与技术无关；越靠外越易变。跨越边界时，靠接口把控制流与依赖流反向解耦。

→ P · 01
依赖倒置

高层不依赖低层，两者都依赖抽象。接口由内层定义、外层实现——这是三种架构共同的机械原理。

◎ P · 02
可测试性

领域不绑定框架，便能在没有数据库、没有 Web 服务器的情况下，用纯粹的单元测试验证业务规则。

⇄ P · 03
延迟决策

数据库、消息中间件、UI 框架都是「细节」。好架构让你把这些选择推迟到最后一刻，且随时可换。

06 Worked Example · aemeath 实战

一个真实 Rust 项目的 DDD 重构REAL REFACTOR · FROM CODE TO BOUNDED CONTEXTS

电商案例是从零建模；这一节反过来——一个已有 16 crate 的 AI 编程助手，代码先行、边界模糊，如何用 DDD 倒推出清晰的限界上下文。 五个阶段贯穿三条经验：不按 crate 划 BC、复杂度 ≠ BC、用领域本质回答"要不要状态机"。

I

战略前置 · 盘点事实

DDD · 战略

画任何边界前，先把代码事实摸清，拒绝拍脑袋。

  • 盘 16 crate + SDK 契约 + AgentClient 入站端口 + 持久化现状
  • 精读现有 8 份设计文档，逐份标 Current vs Target
  • 发现核心 runtime 里塞了 11 组能力（compact / reflection / cost / scheduler…）
产出 · 一张「能力原子」清单，作为划界原料
II

战略设计 · 按能力聚类划 BC

DDD · 战略

拒绝「一个 crate = 一个 BC」，从领域能力重新聚类。

  • 划子域：核心 / 支撑 / 通用
  • 主动挖遗漏：Context Management · Reflection · Scheduler · Update 一开始都没列
  • 暴露问题：Scheduler 是死代码、Update 命名错位应为 Version Control
产出 · 15 个 BC 候选 + 主动暴露的遗漏项
III

战略设计 · 两把判据辨边界

DDD · 战略

用两把尺子切开「看似要独立」的能力。

  • 尺一 · 有无独立聚合+不变量+生命周期？ → Task 是 BC，Interaction 不是
  • 尺二 · 定义契约 vs 履行契约？ → SDK 是核心域契约，clients 是履行者
  • Loop 很复杂但无独立聚合 → 是模块，不是 BC
产出 · 边界收敛，伪 BC 被拒之门外
IV

战略设计 · 简化与收敛

DDD · 战略

每个「要不要状态机 / 要不要 durable」都用领域本质回答。

  • Session 的「状态」全是投影或 IO 动作 → 单状态机，只留 AgentRun（内存态）
  • Session 主体是对话历史 → 并入 Context Management，不独立
  • 人在环 CLI 崩溃从头开始 → 去掉 durable invocation
产出 · 从 14 BC + 双状态机 收敛为 15 BC + 单状态机
V

架构落地 · 上下文地图 + 落文档

六边形整洁架构

把边界翻译成端口关系与目录结构。

  • 集成关系：C/S · ACL · Pub/Sub · OHS · PL · SK
  • 交付层（clients）+ Future 预留（Server 传输透明）；Workflow 降为支撑域
  • 落 docs/design 三层：01-system 总体 → 02-modules 模块 → 03-engineering 横切

VI

迭代与修正 · 边界不是一次画对

DDD · 战略

真实项目里，命名和边界都要反复用判据检验、推倒重来——初稿不是终稿。

  • 命名三次迭代：Chat → AgentRun → Run（对齐 OpenAI Thread / Run / Run Step）
  • BC 归类修正：Workflow 一度进核心域，再用判据降为支撑域（复杂度 / 概念 ≠ 核心）
  • 只写目标态：删掉所有「当前代码落点」，避免代码演进后文档过时造成混淆
产出 · 15 BC 定稿（1 核心 + 8 支撑 + 6 通用），单状态机 Run
07 Disambiguation · 概念辨析

长得像，却各司其职TWO RULERS CUT THE FOG

这些混淆几乎都能归到两把尺子：「它有没有身份？」与「它是定义契约还是履行契约？」 抓住这两把尺，大部分歧义自解。

A · 同名陷阱 · 一词两义，最坑
EntityDDD✕EntityClean
DDD · 一个对象有唯一标识的领域对象，如 Order、Customer。对立面是值对象。
Clean · 一整环最内层的企业级业务规则圈，内部可容纳 DDD 的 Entity/VO/Aggregate。
DDD 的 Entity 是名词（一个对象），Clean 的 Entity 是地名（最里那一环）。
领域服务✕应用服务
Domain Service装不属于单个聚合的业务规则，如 PricingService。含逻辑。
Application Service只做编排：调聚合、仓储、管事务。不含业务规则。
应用服务是导演（只调度），领域服务是专家（只贡献判断）。
Repository✕DAO
Repository内存集合的错觉，面向聚合根，接口在 domain。
DAO一张表的 CRUD 映射，面向数据，纯技术层。
Repo 让领域以为数据在内存里；save(order) 会连同 OrderItem 一起落盘。
B · 结构对 · 谁是谁
聚合✕聚合根
Aggregate共享一致性边界的一组对象：Order + 全部 OrderItem。
Aggregate Root聚合唯一入口与守门人：Order。外部只能经它操作。
聚合是范围，聚合根是门。
实体✕值对象
Entity在乎「是不是同一个」，靠 ID 区分。两个同额 Order 仍不同。
Value Object不在乎身份，不可变，按属性相等。Money(100,CNY) 可互换。
唯一判据——你在乎它的身份吗？Address 常被误建成 Entity。
端口✕适配器
Port · 契约由核心定义的接口：「我要能扣款的东西」→ PaymentGateway。
Adapter · 翻译官具体实现：StripeAdapter 把 Stripe API 翻成 PaymentGateway。
一个端口可挂多个适配器（Stripe/支付宝/Fake）。端口不变，适配器随便换。
驱动端口✕被驱动端口
Driving · 主动侧别人调你：Controller → UseCase。接口你实现。
Driven · 被动侧你调别人：Service → Repository。接口你声明、适配器实现。
看控制流从谁发起：发起方在左，被调方在右。
子域✕限界上下文
Subdomain · 问题空间业务客观分成订单/库存/支付，与技术无关。
Bounded Context · 解空间你在软件里划出的模型边界。
子域是天生的，上下文是设计出来的；理想一对一，现实常错位。
上下文✕限界上下文
Context · 语境通用词：词与模型赖以成立的语境，隐含、无边界、会蔓延。
Bounded Context · 限界划明边界的语境，内部统一语言严格一致，跨界含义可不同。
差别全在限界 Bounded——有没有那条边界。日常说的「上下文」多半就是它的简称（如 Context Map）。
C · 同一件事的三种叫法
用例 = 应用服务 = 驱动端口 · 同一段编排代码
CleanUse Case / Interactor
HexagonalDriving Port + 其实现
DDDApplication Service
看到这三个词并列，先默认它们在说同一层东西。
D · 三种「同心圆」架构的区别
六边形 ✕ 洋葱 ✕ 整洁 · 同一座山的三张地图
Hexagonal · 2005强调对称插拔（端口 / 适配器）。
Onion · 2008强调多层同心、领域居中。
Clean · 2012一句「依赖规则」统摄 + 控制流反转。
别纠结用哪个。工程上常混用：整洁的分层 + 六边形的端口 + DDD 的内涵。
E · 建模质量 · 一字之差，天壤之别
充血模型✕贫血模型
Rich · 充血order.pay() / cancel() 自守不变量，行为与数据在一起。
Anemic · 贫血实体只有 getter/setter，逻辑全在 Service——DDD 反模式。
实体是一袋数据、规则全在 Service，你写的就是贫血模型。
值对象✕DTO
Value Object领域内、不可变、有行为（Money.add）、守规则。
DTO跨边界搬数据、可变、无行为、纯字段袋（PlaceOrderRequest）。
都无标识，但一个守规则、一个只搬运；别让 DTO 泄进领域层。
防腐层✕适配器
ACL · 防腐层翻译模型 / 概念，防外部世界观污染领域。
Adapter · 适配器翻译技术格式（HTTP ↔ 对象）。
防腐层是适配器的「加强用途」：由适配器 + 转换器实现，多一层概念翻译。

两把尺收束全部：有没有身份（Entity/VO、子域/上下文）· 定义契约还是履行契约（端口/适配器、领域服务/应用服务）。
