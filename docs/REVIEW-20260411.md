# aemeath 项目 Code Review 报告

> 审查日期：2026-04-11

## 项目概览

aemeath 是一个 Rust 编写的 AI 编程助手 CLI 工具，支持多 LLM Provider、工具调用、MCP 协议、TUI 界面。项目采用 workspace 结构，分为 4 个 crate：

| Crate | 职责 |
|-------|------|
| `aemeath-core` | 核心数据结构、配置、session、权限、token 估算 |
| `aemeath-llm` | LLM Provider 抽象层（Anthropic/OpenAI/Ollama） |
| `aemeath-tools` | 工具实现（Bash、文件读写、搜索等 ~30 个工具） |
| `aemeath-cli` | CLI 入口，含 TUI 和 REPL 两种交互模式 |

---

## 一、安全问题

### 🔴 高危

#### H1. API Key 明文存储在 config.json

`aemeath-core/src/config.rs` 中 `ProviderModelsConfig.api_key` 是纯文本 String，直接序列化到 `~/.aemeath/config.json`。`main.rs:487-494` 读取时还有一个更危险的问题——会遍历所有 provider 取第一个非空 api_key 作为 fallback，可能用错 provider 的 key。

```rust
// main.rs L487-494
// 甚至会尝试任意 provider 的 key 作为 fallback!
for (_, pcfg) in &cfg.models.providers {
    if !pcfg.api_key.is_empty() {
        return pcfg.api_key.clone();  // 遍历所有 provider 取第一个
    }
}
```

**建议**：使用系统密钥链（macOS Keychain / Linux secret-service）或至少加密存储。移除 fallback 遍历逻辑。

---

#### H2. FileWrite/FileEdit 缺乏路径遍历防护

`file_read.rs` 有完善的 `validate_and_normalize_path` + `has_traversal_attempt` 检查，但 `file_write.rs` 和 `file_edit.rs` **完全没有路径验证**，LLM 可被间接注入诱导写入 `~/.ssh/authorized_keys`、`~/.bashrc` 等敏感路径。`file_edit.rs` 还会自动创建不存在的父目录。

```rust
// file_write.rs L36 — 直接使用原始路径，无任何边界检查
let path = Path::new(file_path);
```

**建议**：将 FileRead 的路径验证提取为公共函数，所有文件操作工具统一使用。

---

#### H3. allow_all 模式下 Shell 注入检查被完全跳过

`bash.rs:250-257`：当 `allow_all=true` 时，`check_shell_injection` 不执行。而 `allow_all` 可通过 `config.json` 静默启用（无需用户交互确认），攻击者如果能写入 config.json，就能绕过所有 shell 注入防护。

```rust
// bash.rs L250-257
if !ctx.allow_all {
    if let Some(reason) = check_shell_injection(command) {
        return ToolResult::error(...);
    }
}
```

**建议**：`allow_all` 的启用应要求用户显式确认，或至少在启动时打印醒目警告。

---

### 🟠 中危

#### M1. Bash 命令注入过滤器存在多个绕过向量

- `curl -s`、`wget -q` 被标记为 `readonly`，但 `curl` 可以上传数据（`-T`、`-d`、`-o`）
- `python -c`、`node -e`、`ruby -e` 被标记为 `readonly`，但它们可以执行任意代码
- `is_safe_chain_command` 的逻辑与注释矛盾（注释说"总是危险"，代码却有条件放行）
- 管道分割 `split(|c| c == '&' || c == '|')` 会把 `&&` 和 `||` 错误地多分割

**建议**：重写命令解析器，使用适当的 shell 语法解析而非简单的字符串分割。将 `curl`/`wget`/`python -c` 等从 `readonly` 列表移除。

---

#### M2. MCP 客户端执行任意命令无沙箱

`mcp.rs:69` 直接从 `.mcp.json` 读取并执行任意命令+参数+环境变量。项目级 `.mcp.json` 位于工作目录中，clone 的恶意仓库可自动执行代码。`env` 字段还可注入 `LD_PRELOAD`、`PATH`。

```rust
// mcp.rs L69
let mut cmd = Command::new(&config.command);  // 执行任意命令
cmd.args(&config.args)                        // 任意参数
    .stdin(Stdio::piped())
    .stdout(Stdio::piped())
    .stderr(Stdio::null());
for (k, v) in &config.env {
    cmd.env(k, v);                            // 任意环境变量
}
```

**建议**：连接 MCP server 前需用户确认；不自动加载项目级 `.mcp.json`；过滤危险环境变量。

---

#### M3. Session 文件未限制权限

Session 文件可能包含敏感对话内容，写入磁盘时使用默认 umask 权限，多用户系统上其他用户可读。

**建议**：使用 `OpenOptions::new().mode(0o600)` 限制文件权限。

## 二、架构与代码质量问题

### 🔴 严重

#### A1. Session 类型重复定义

`state.rs` 中有 `InternalSession`，`session.rs` 中有 `Session`，两者功能大量重叠：

- 都有 `id`、`cwd`、`messages`、`created_at`、`updated_at`
- 都有 `save_session` / `load_session` 函数
- 都有独立的 `new_session_id()` 实现，且**生成格式不同**（`state.rs` 生成 hex 格式，`session.rs` 生成纯数字时间戳）
- 都有独立的 `validate_session_id`（实现相同，但重复了）

`state.rs` 中的 `Session` 已标记为 `#[deprecated]`，但 `AppState` 仍然使用 `InternalSession`，与 `session::Session` 并行存在。

**建议**：统一为单一的 `Session` 类型，删除 `InternalSession` 和 `AppState` 中重复的 session 管理逻辑。

---

#### A2. ConfigManager 与 AppState 功能重叠

`ConfigManager` 和 `AppState` 都管理配置和 session 持久化，都有 `load`/`save`/`get`/`update` 方法。`AppState` 实际上只在 REPL 中被创建但没有真正使用（TUI 模式不使用它）。

**建议**：只保留 `ConfigManager`，移除 `AppState`。

---

#### A3. 错误处理不统一

- `aemeath-core/src/error.rs` 定义了完善的 `AemeathError` 枚举，但**几乎没有模块使用它**
- 几乎所有函数返回 `Result<T, String>`，放弃了结构化错误处理
- `aemeath-llm` 有自己独立的 `LlmError`，与 `AemeathError` 没有转换关系
- `ConfigManager`、`AppState`、`session` 模块全部用 `String` 错误

**建议**：统一使用 `AemeathError`，各模块实现 `From<AemeathError>` 转换。

---

#### A4. PermissionMode 重复定义三次

- `state.rs::PermissionMode`
- `config.rs::PermissionModeConfig`
- `permission.rs::PermissionMode`

三个枚举的变体完全相同（Ask / AutoRead / AllowAll），但没有共享定义。

**建议**：统一到 `permission.rs`，其他模块引用。

---

### 🟡 一般

#### A5. LLM Provider 间大量代码重复

`OllamaProvider` 和 `OpenAICompatibleProvider` 的 `convert_messages`、`convert_tools`、`convert_system_to_message` 约 100+ 行代码完全重复。

**建议**：提取到共享模块（如 `openai_format.rs`），Ollama provider 复用 OpenAI compatible 的转换逻辑。

---

#### A6. `format_tokens` 函数重复定义

`token_estimation.rs` 和 `cost.rs` 各有一个 `format_tokens` 函数，逻辑基本相同。

**建议**：提取到公共模块，两处引用。

---

#### A7. ISO 时间格式手动实现

`session.rs` 的 `now_iso()` 和 `main.rs` 的 `current_date()` 都手动实现了日期计算（约 20 行重复代码），容易出错。项目已有 `chrono` 依赖（在 aemeath-tools 中），应统一使用。

**建议**：在 `aemeath-core` 的 Cargo.toml 中添加 `chrono` 依赖，统一时间处理。

---

#### A8. CostTracker 使用同步 I/O

`CostTracker` 使用 `std::fs` 同步文件操作，但它在 async 上下文中被调用。

**建议**：改用 `tokio::fs` 或使用 `tokio::task::spawn_blocking` 包装同步调用。

---

## 三、LLM Provider 实现问题

### 🔴 严重

#### L1. Stream Idle Timeout 永远不会触发

`stream.rs` 和其他两个 provider 的流解析器中，`tokio::time::sleep(STREAM_IDLE_TIMEOUT)` 在每次 `lines.next_line()` 返回后重新创建，所以 90 秒超时实际上永远不会触发——只要每 90 秒内收到任意一行就重置。

```rust
// stream.rs L39-57 — 每次循环创建新的 sleep Future
let line = tokio::select! {
    biased;
    _ = cancel.cancelled() => { ... }
    _ = tokio::time::sleep(STREAM_IDLE_TIMEOUT) => { ... }  // 每次循环重新创建！
    result = lines.next_line() => { ... }
};
```

**建议**：在循环外用 `Instant` 跟踪上次收到有效数据的时间，循环内手动判断。

---

#### L2. Stall 检测是完全的死代码

```rust
// stream.rs L63-66
if gap > STALL_THRESHOLD {
    // Stream stall detected — silently ignored
}
```

检测到问题但什么都没做——没有日志、没有通知、没有恢复。给了开发者错误的信心。

**建议**：至少调用 `handler.on_error()` 通知上层，或记录 `tracing::warn!`。

---

#### L3. AnthropicProvider 中 timeout_secs 未被使用

`anthropic.rs:60,76`：`timeout_secs` 字段存储了值但 HTTP client 使用硬编码的 120 秒超时，`with_timeout_secs` 方法不存在。

**建议**：使用 `timeout_secs` 字段构建 HTTP client，或添加 `with_timeout_secs` 方法。

---

### 🟡 一般

#### L4. OpenAI 流式 `on_tool_use_start` 重复调用

delta 阶段和最终汇总阶段各调用一次，导致 UI 重复通知。

**建议**：只在最终汇总阶段调用 `on_tool_use_start`，或用标记防止重复。

---

#### L5. JSON 解析失败被静默吞掉

SSE 事件 JSON 解析失败时无任何日志，调试困难。

```rust
// stream.rs L83-86
let event: StreamEvent = match serde_json::from_str(data) {
    Ok(e) => e,
    Err(_) => continue,  // 完全静默
};
```

**建议**：在 `Err` 分支中加 `tracing::debug!` 记录失败的原始数据。

---

#### L6. Tool JSON 解析失败默认空对象

流被截断时 `unwrap_or(Value::Object(Map::new()))` 导致 tool 执行收到错误的空参数，无任何警告。

**建议**：在 `unwrap_or` 分支中添加 `tracing::warn!`。

---

#### L7. Usage 缺少 cache token 字段

Anthropic 的 `cache_read_input_tokens`、`cache_creation_input_tokens` 被忽略，影响计费准确性。

**建议**：扩展 `Usage` 结构体，添加 cache 相关字段。

---

#### L8. `stream_options` 不兼容非标准 OpenAI API

部分 OpenAI 兼容 API 不支持 `stream_options`，可能返回错误或忽略。无降级处理。

**建议**：根据 `Provider` 类型决定是否发送 `stream_options`。

---

#### L9. Ollama 网络错误不重试

Ollama provider 对超时错误会重试，但对其他网络错误直接返回。与 Anthropic/OpenAI 的行为不一致。

---

## 四、其他问题

#### O1. `#[allow(dead_code)]` 标注较多

多个结构体和方法带有 `#[allow(dead_code)]`，暗示有未使用的代码路径，应清理或移除。

---

#### O2. `Task::is_blocked` 每次调用 clone 整个 tasks HashMap

对于频繁调用，这是不必要的性能开销。

**建议**：改为持锁遍历，或使用更细粒度的锁。

---

#### O3. `TaskStore::create` 和 `create_with_priority` 代码几乎完全重复

应提取公共逻辑。

---

#### O4. `scheduler.rs` 的 `complete_task` 中失败任务重置为 Pending 而非 Failed

语义上不合理，应新增 `Failed` 状态。

---
